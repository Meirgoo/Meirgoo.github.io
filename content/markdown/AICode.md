## 程序员会梦到AI抢饭碗吗？

我的观点是，不会。

昨天看到一个有意思的评论，是在一个贩卖AI课程的视频下面说的，视频内容还是老一套，贩卖焦虑吹捧AI巴拉巴拉的，视频看多了大致也会有个基本印象。那条评论原文我没存，大意就是现在的媒体卖课本质上就是在一个贩卖焦虑的时代贩卖赎罪券，好像一个赛博学习的宗教，AI就是主，一群沉淀不下来的人买了赎罪券，好像就学到了知识，好像拿到了通往未来的钥匙。

还有人对AI很失望，觉得AI就是个大型百度。

这么说也未尝不可。现在的AI还是基于算法基础使用生成的模型和已知素材进行归纳总结返回你要的结果，从实质上来说和搜索引擎确实没有根本性的区别，只不过对外他可以有一个对话框，可以以一种人类的交流方式去完成这个搜索过程，并且对信息的归纳总结能力远强于一般人。

现在的AI没有足够的创造性，也不能证明真正理解了文字内容，确实只是算法构成的一种假象，但是这种信息搜集能力能直接的辅助学习的过程，如果我去看官方的开发文档会有什么问题？版本问题，表述问题，翻译问题等等，谷歌的开发文档浅层条目还有已经淘汰技术的相关内容，可以用但是IDE会告诉你不建议使用或者已废弃，那就要继续深挖，带来了一定的试错成本。

还有文档用专业的表述进行，不得不说对初学来说确实有难度，deepseek深度思考返回的结果加几个示例能在几个小时内辅助大致理解一个没用过的技术，效率比纯自己悟要高不少。

在ChatGPT刚出来的时候，有不少声音说程序员要被取代了，还搬出了著名的悖论，程序员做的工作本质上是为了淘汰自己，在使用deepseek这个目前理论最强AI的过程中，我觉得离完全取代程序员尚有距离，不如说先被取代的可能是低端教育行业以及纯对话的服务行业，例如客服等等，一切工程行业并不是由纯粹的几个技术知识构成的，或许在未来的某一天AI可以，但绝不是现在。

## AI的局限

最近看了一个程序员整活视频 [Java特烦恼](https://www.bilibili.com/video/BV1QTwUeJEqD/?spm_id_from=333.337.search-card.all.click&vd_source=9961f29dcf00ebc80afadcda2a74f2cf)。

不得不说，删程序依赖着实有点狠。

如果向非专业人士解释，依赖就是比个人写的代码在逻辑上更加底层的封装好的东西，程序员都听过一句话叫“不要重复造轮子”，引用依赖基本是绝大多数用途下程序的必有的一步。

昨天我在写一个组建的时候，需要结合compose和webview两个技术，但是compose没有原生的兼容组件，我就让AI帮我搓一个示例让我学一下，然后AI就给我指出了几个依赖给出了相关代码，那几个依赖是这样的。


```
// 必需依赖
implementation "androidx.compose.ui:ui:1.6.3"
implementation "androidx.compose.foundation:foundation:1.6.3"
implementation "androidx.compose.material:material:1.6.3"

// WebView 支持
implementation "androidx.webkit:webkit:1.10.0"
implementation "com.google.accompanist:accompanist-webview:0.34.0" // 可选增强

// 颜色处理
implementation "androidx.compose.ui:ui-graphics:1.6.3"
```

deepseek后续给出的代码我相信没有太大问题，为什么是相信而不是确定呢？因为这段代码根本运行不了，AI所给出的代码是过时的，相关内容已被废弃无法使用。

```
    private val webViewState: WebViewState = WebViewState()
```

从开始的一行代码就开始了提示，该方法的依赖已于23年8月份停止维护，deepseek已经是相当深度的检索能力了，偶尔也会出现这种引用无法引用的依赖的错误，其余ai虚空捏造也不在少数，像deepseek使用ComponentActivity类的方法套在Activity类上也出现过，这些还都是比较小的问题。

在项目创立初期，我是用deepseek协助我设计结构和流程，具体实现上deepseek对于相关文件的解析只是维持了最低限度的能力（而且还有待证实），整体架构的模块化扩展性都不足，还有大片留白。

或许运气好AI能顺着自己定义的架构搞出来一套能直接运行的代码，去实现不懂代码也能写软件，但目前仍做不到那一步，AI可以协助某一个具体功能的粗略实现，但是在其他方面都不足以胜任。

往小了说，系统功能需要微调定制，或许有特殊需求，简单繁复且细致的调整并不适合AI应用场景，有一定代码能力大都能做到，还能更称心。

往大了说，一个软件整体的功能代码实现只是一部分，真正深度理解软件需求和开发方向，去把控整体架构，仍然是AI做不到的。

## AI？一门生意

昨天看到一篇公众号文章的论点很有意思，说软件开发的核心从来不在代码和实现，而是对于需求和痛点的理解，如果能深入理解拆解原子级的痛点，那就是市场上最为稀缺的“深度的价值”，而现有的大量AI产品本质上并没有实际价值，只是“API”小贩，只是在别人已有产品的基础上套个壳然后坐收门票钱。看到有人说某AI绘图接口涨价导致某网站收益模型崩坏，部分业务直接关闭，也不知道是不是我前段时间想玩的那个知名AI绘图网站。

某些网站也有拒绝AI产出内容的苗头，我觉得那句话挺对的，人与人之间的相互理解和需求的共情，是任何算法都复制不了的资产。
